04 · Git Pipeline

GitOps Deploy Pipeline

prod = Windows container (CI build จาก Dockerfile.windows, nanoserver). ทุกอย่างอัตโนมัติ: GitHub Actions build → Trivy gate → push image → commit tag เข้า gitops repo เอง → server pull + blue-green. deploy เกิดเมื่อ compose-hash เปลี่ยน (มาจาก git pull gitops เท่านั้น) · network = nat

Security gate

Trivy ×2

fs (deps) + image (os/library) · HIGH/CRITICAL · --ignore-unfixed · --exit-code 1 = build fail ไม่ push (มี SARIF report-only แยกด้วย)

Image tag

<version>-<commit>

จาก package.json version + short SHA — กัน tag ชน + traceable

Auto-bump

github-actions[bot]

regex แทน tag ใน docker-compose.template.yml ของ gitops repo → commit + push

Blue-Green Deploy — ไม่มี downtime

deploy.ps1 ดึง image แล้วขึ้น slot สำรอง, health ผ่านค่อยสลับ ARR. pull fail (offline) ถูก tolerate — up -d รันจาก image ที่มีอยู่แล้ว

05 · Setup

Setup Server + เพิ่ม Site ใหม่ (White-Label)

หนึ่ง Windows Server รันได้ หลายแบรนด์/ลูกค้า (site) พร้อมกัน — แยกกันด้วย environment variable ของ NSSM service. ทุก site clone gitops repo เดียวกัน ความต่างมาจาก AppEnvironmentExtra ล้วน ๆ

Shared — ติดตั้งครั้งเดียว/เครื่อง

Redis (NSSM, แยกด้วย REDIS_DB) · Mailpit (NSSM) · Docker network gitops_default (driver nat) · SQL Server instance เดียว · IIS / ARR proxy

Per-site — ทำซ้ำต่อแบรนด์

GitOps dir C:\gitops-<brand> · NSSM <brand>-gitops-deploy · DB <brand>_db · IIS site <BRAND> · containers <brand>-blue/green · ports ห้ามชนกัน

Windows NAT gotcha: ใช้ network เดียว gitops_default ทุก site — สร้างหลาย NAT network ไม่ได้ HNS จะค้าง. และ host.docker.internal ไม่ resolve บน Windows container → DB_HOST ต้องเป็นชื่อเครื่อง / NAT-gateway IP

Flow วันส่งมอบ (server มีเน็ตตอน setup)

setup-site.ps1 — กรณี B ทั้งชุด (idempotent)

  1. เช็ก port ว่าง — ถ้า port โดน container brand เดียวกันถือ (re-run) = OK; โดน process อื่น = throw
  2. GitOps dir — clone gitops repo, ฝัง -GitHubPat ใน remote URL ให้ SYSTEM pull ได้, สร้าง upload + log dir
  3. Secret files — เขียน .env.docker, สร้าง .env.secrets + เติม BASE_URL/DATABASE_NAME/REDIS_DB/JWT อัตโนมัติ (ที่เหลือกรอกเอง)
  4. Docker network — สร้าง gitops_default (nat) ถ้ายังไม่มี
  5. Database (เฉพาะ -CreateDatabase) — sqlcmd IF DB_ID IS NULL CREATE DATABASE · collation Thai_CI_AS
  6. IIS site + app pool — pool No Managed Code, host header -Domain :80, + HTTPS 443 SNI ถ้ามี -CertThumbprint
  7. NSSM deploy service — <brand>-gitops-deploy รัน deploy.ps1 + env ครบ (COMPOSE_PROJECT_PREFIX, ports, ...)
  8. Start — เฉพาะ -StartService และ .env.secrets ไม่มี placeholder เหลือ
verify: doctor.ps1 -Brand <brand> (read-only) เช็ก tooling · IIS/ARR · services · shared ports (1433/6379/1025/8025/80/443) · firewall · network · active slot + /api/v1/health (ok:true). exit 0 = ไม่มี FAIL · -Brand ต้องตรง ^[a-z0-9][a-z0-9-]{1,20}$

08 · Offline

ลูกค้า Offline (first-setup-online แล้วตัดเน็ต)

เครื่อง server มี เน็ตตอน setup ครั้งแรก → docker compose pull ทำงานปกติ ไม่ต้อง docker save/load. หลัง handover ตัดเน็ต → pull fail ทุก tick แต่ harmless

online ทั่วไปลูกค้า offline
SQL Servernative บนเครื่องnative (IT ลูกค้าลงให้ก่อน)
ตอน setup แรกมีเน็ต → pull web/apiมีเน็ต → pull ได้ (ทำ §05 ครบช่วงนี้)
หลัง setupมีเน็ต → deploy ใหม่ได้ตัดเน็ต → pull fail แต่ container รันต่อจาก image เดิม
ทำไม่ได้หลังตัดเน็ต—deploy web/api ใหม่ + Android OTA
ทำให้ครบ "ช่วงยังมีเน็ต": รัน §05 → ปล่อย deploy loop pull image ครบ + blue-green switch สำเร็จอย่างน้อย 1 รอบ + doctor.ps1 ผ่านหมด ก่อน ตัดเน็ต. deploy ใหม่เกิดเฉพาะตอน compose-hash เปลี่ยน (มาจาก git pull) → offline = ไม่ redeploy เอง
